# HEBBS — Positioning & Branding Guide

**Primary:** hebbs.ai

## Name Origin

HEBBS derives from **Hebbian learning** — Donald Hebb's 1949 neuroscience principle: *"neurons that fire together wire together."* This is exactly what HEBBS does for AI agents: memories that get recalled together strengthen together; memories that stop being useful decay away.

The name follows the playbook of the best infrastructure projects:

| Project | Name Origin | Letters |
|---------|-------------|---------|
| Redis | **RE**mote **DI**ctionary **S**erver | 5 |
| Kafka | Franz Kafka (literary) | 5 |
| etcd | `/etc` + **d**istributed | 4 |
| NATS | Neural Autonomic Transport System | 4 |
| **HEBBS** | **Hebb**ian learning (neuroscience) | 5 |

Short, punchy, scientific pedigree. Sounds like infrastructure, not a startup.

## Category

HEBBS creates a new category: **cognitive memory engine**.

Not a database. Not a vector store. Not a framework plugin. A *primitive* — the same way Redis is a primitive for caching, Kafka is a primitive for streaming, and Postgres is a primitive for relational data.

## Positioning Statement

> HEBBS is the memory primitive for AI agents. It replaces the patchwork of vector databases, key-value stores, and graph databases that agent developers stitch together today with a single, native engine — purpose-built for cognition, not storage.

## Competitive Positioning

### The Frame

Every comparison should make the alternative sound like it belongs to a previous era:

| vs. | Their story | Our story |
|-----|------------|-----------|
| Vector DBs (Qdrant, Pinecone) | "We search embeddings" | "They search. We *remember*." |
| Redis / KV stores | "We cache fast" | "They cache. We *learn*." |
| Knowledge Graphs (Neo4j) | "We store relationships" | "They link. We *understand*." |
| Framework memory (LangChain, Mem0) | "We wrap 4 services" | "They stitch. We ship *one binary*." |
| Conversation history | "We store chat logs" | "They scroll. We *consolidate*." |

### The Redis Analogy

This is the most powerful positioning tool. Use it everywhere:

> Redis was to Memcache what HEBBS is to vector databases.
>
> Redis didn't just make caching faster — it introduced data structures (sorted sets, pub/sub, streams) that changed what caching could *mean*.
>
> HEBBS doesn't just make recall faster — it introduces cognitive operations (decay, reinforcement, consolidation, causal reasoning, analogical transfer) that change what memory can *mean*.

## Brand Voice

1. **Confident, not arrogant.** State what we are. Don't trash competitors by name. Let the benchmarks speak.
2. **Technical, not jargon-heavy.** Use precise terms (HNSW, B-tree, ONNX) but always explain the *why*.
3. **Infrastructure-grade tone.** This is not a "delightful developer experience" play. This is "your agent's memory will not lose data, period." Think Cloudflare's blog voice.
4. **Numbers over adjectives.** Don't say "blazing fast." Say "384 nanoseconds." Don't say "massive scale." Say "10 million memories, 8ms p99."

## Visual Identity Notes

- **Tone:** Infrastructure. Think RocksDB, Cloudflare, Fly.io — not Notion or Linear.
- **Color palette:** Dark backgrounds, monospace type for code/numbers, accent color for emphasis.
- **Logo direction:** The name "HEBBS" in a monospace or geometric sans-serif. No icons, no mascots. The name is the brand. Infrastructure projects don't need cute logos — they need recognition.
- **Hero visual:** Architecture diagram or a single terminal screenshot showing the speed. No stock photos. No abstract gradients. Show the thing.
