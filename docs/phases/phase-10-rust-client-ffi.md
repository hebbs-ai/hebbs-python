# Phase 10: Rust Client SDK and FFI -- Architecture Blueprint

## Status: PENDING

---

## Intent

Phase 10 bridges the gap between HEBBS-the-engine and HEBBS-as-a-building-block-for-other-systems. Until now, the only ways to use HEBBS are: (a) link `hebbs-core` directly as a Rust library dependency, which pulls in RocksDB, ONNX Runtime, and 40+ transitive dependencies, or (b) construct raw gRPC calls against `hebbs-server` using the tonic-generated stubs from `hebbs-proto`, which exposes protobuf types (`prost` structs with `i32` enums, `Option<prost_types::Struct>` for context, `Vec<u8>` for memory IDs) that are awkward, unergonomic, and leak the wire format into application code.

Neither path is acceptable for external consumers. A Rust developer building an agent framework should not need to understand protobuf encoding to call `remember()`. A Python developer embedding HEBBS via PyO3 should not need to run a server process. A C++ robotics team integrating HEBBS into their control loop should not need the Rust async runtime.

Phase 10 delivers two crates that solve these problems:

**`hebbs-client`** -- a thin, async Rust client library that wraps the gRPC stubs with ergonomic Rust types, connection management, retry logic, and timeout handling. The developer experience is: `let client = HebbsClient::builder().endpoint("localhost:6380").build().await?;` followed by `client.remember("the customer prefers email contact", 0.8).await?`. The proto layer is invisible.

**`hebbs-ffi`** -- a C-ABI foreign function interface that links `hebbs-core` directly into a shared library (`.so` / `.dylib` / `.dll`). No server, no network, no async runtime. The developer experience is: `hebbs_open("./agent-memory")` returns an opaque handle, `hebbs_remember(handle, content, len, importance)` writes a memory, `hebbs_close(handle)` shuts down. The C header file is generated by `cbindgen`. This is the foundation that Phase 11 (Python SDK via PyO3) and every future language binding builds on.

The decisions made here are load-bearing for five downstream concerns:

- **Phase 11 (Python SDK):** PyO3 wraps the FFI layer for embedded mode and the client SDK for server mode. The FFI handle pattern, error reporting mechanism, and memory ownership rules determine whether the Python SDK is safe and idiomatic or a minefield of segfaults and leaked memory. Every unsafety boundary, every ownership transfer, every error propagation path must be airtight before Python wraps it.

- **Phase 12 (Testing and Benchmark Suite):** The client SDK becomes the standard way to write integration tests against a running server. Test ergonomics depend entirely on the client API surface. If `remember()` requires constructing a `RememberRequest` proto message, tests will be verbose and fragile. If it accepts `(&str, f32)`, tests will be clean and readable. The benchmark suite uses the client SDK for server-mode benchmarks, so any overhead the client adds (serialization, connection management) becomes part of the measured latency.

- **Phase 17 (Edge Mode and Sync):** Edge deployments use the FFI layer for in-process HEBBS. The FFI must support the full lifecycle: open database, configure decay and embedding, perform all 9 operations, close cleanly. If the FFI layer cannot express configuration, edge mode requires a separate code path. The FFI configuration model must accommodate edge-specific parameters (384-dim embeddings, local LLM, aggressive decay) without structural changes.

- **Phases 18-19 (TypeScript and Go SDKs):** These SDKs connect to the server via gRPC stubs generated from the same `.proto` files. They do not depend on `hebbs-client` directly. However, the ergonomic patterns established in Phase 10 -- how memory IDs are represented, how errors are categorized, how subscribe streams are consumed -- set the precedent that TypeScript and Go SDKs should follow for API consistency across languages. The client SDK is the reference implementation.

- **Phase 16 (Documentation):** The client SDK is the first thing a Rust developer sees in the "Getting Started" guide. The 50-line integration example in the README (Principle 10: "first integration in < 50 lines") is written against `hebbs-client`, not `hebbs-proto`. The API names, method signatures, and error messages in this crate are the public face of HEBBS for the Rust ecosystem.

## Scope Boundaries

### What Phase 10 delivers

- `hebbs-client` crate: async Rust client SDK over gRPC
  - Builder pattern for client construction with endpoint, timeout, retry, and TLS configuration
  - All 9 HEBBS operations as ergonomic async methods accepting Rust-native types
  - `subscribe()` returning a typed async `Stream` of push events
  - Connection management with lazy connect, automatic reconnect on transient failure, and channel health monitoring
  - Configurable retry policy for idempotent operations
  - Client-side timeout enforcement independent of server timeout
  - Type conversion layer: proto types to/from domain types that mirror `hebbs-core` semantics without depending on `hebbs-core`
  - Client-specific error taxonomy mapping gRPC status codes to actionable Rust errors
  - Connection pooling strategy for high-throughput scenarios

- `hebbs-ffi` crate: C-ABI foreign function interface for embedded mode
  - Opaque handle pattern (`HebbsHandle`) wrapping `Arc<Engine>` behind a raw pointer
  - C functions for all 9 operations plus lifecycle (open, close, configure)
  - `cbindgen`-generated C header file (`hebbs.h`)
  - Thread-local error reporting with `hebbs_last_error()` for detailed error messages
  - Explicit memory ownership: every allocated return value has a corresponding `hebbs_free_*` function
  - JSON-based configuration for engine initialization (avoids complex C struct hierarchies)
  - JSON-based structured data exchange for context maps, recall results, and insight lists
  - Subscribe support via callback function pointer registration

- Both crates compile as workspace members without pulling in each other's dependencies
- Both crates pass the same logical integration test suite (same operations, same assertions, different transport)

### What Phase 10 explicitly does NOT deliver

- Python bindings (Phase 11 -- but the FFI layer must be designed to wrap cleanly with PyO3)
- Authentication headers or TLS implementation (Phase 13 -- but the client builder must accept `api_key()`, `tls_config()` methods that store the values and apply them when auth is implemented)
- Load balancing across multiple server endpoints (Phase 13 -- but the client builder must accept `endpoints(vec![...])` as a future extension point)
- Embedded mode with custom storage backends (the FFI always uses RocksDB; swappable backends are an internal concern)
- Async trait boundaries on the FFI layer (FFI is inherently synchronous; async is the client SDK's concern)
- Client-side caching or local index (the client is stateless; all state lives on the server or in the FFI engine)

## Architectural Decisions

### 1. Two Crates, Not One

The temptation is to build a single crate that can operate in both modes: connect to a server over gRPC, or open a local database directly. This "dual-mode" pattern is common in database clients (e.g., SQLite in-process vs. client-server).

**Decision: two separate crates with no shared dependency beyond `hebbs-proto`.**

Rationale:

- **Dependency weight.** `hebbs-client` pulls in `tonic` (gRPC) and `tokio` (async runtime). `hebbs-ffi` pulls in `hebbs-core`, `hebbs-storage` (RocksDB), `hebbs-embed` (ONNX Runtime), `hebbs-index` (HNSW), and `hebbs-reflect`. These are fundamentally different dependency trees. A unified crate forces every consumer to pay the compile time and binary size of both, even if they use only one mode.

- **Compilation firewall.** `hebbs-ffi` compiles as a `cdylib` (shared library). `hebbs-client` compiles as a regular Rust library (`rlib`). These are different crate types with different linking semantics. A single crate cannot be both.

- **API surface divergence.** The client SDK is async (every method returns a `Future`). The FFI is synchronous (every function blocks until completion). Unifying these behind a trait would either force async on FFI consumers (requiring a runtime) or force blocking on client consumers (defeating the purpose of async). The abstraction would hurt both sides.

- **Testing isolation.** Client SDK tests require a running `hebbs-server`. FFI tests do not. Mixing them complicates CI pipelines.

The two crates share the same logical API surface -- same operation names, same parameter semantics, same result shapes -- but they are independently compiled, independently versioned (within the workspace), and independently distributed.

### 2. Client SDK Domain Types -- The Conversion Layer

The gRPC stubs in `hebbs-proto` generate prost types: `RememberRequest` with `String` content, `Option<prost_types::Struct>` context, `Vec<Edge>` edges where `Edge.edge_type` is `i32` (not an enum). These types are correct for the wire but hostile for application code.

The `hebbs-core` crate has ergonomic Rust types (`RememberInput`, `RecallInput`, `Memory`, `MemoryKind` enum, `RecallStrategy` enum) but depending on `hebbs-core` from the client SDK would pull in RocksDB and ONNX Runtime -- exactly what the client SDK exists to avoid.

**Decision: `hebbs-client` defines its own domain types that mirror `hebbs-core` semantics, with conversion to/from proto types implemented internally.**

These types live in a `types` module within `hebbs-client`. They look identical to the `hebbs-core` types from the developer's perspective -- same field names, same enum variants, same validation rules -- but they are independently defined. The conversion layer (proto-to-domain and domain-to-proto) is private to the crate.

Why not extract a shared `hebbs-types` crate? Because that crate would need to be a dependency of both `hebbs-core` and `hebbs-client`, which means every change to the types requires coordinated releases. The types in `hebbs-core` are tied to storage serialization (bitcode encoding, `Vec<u8>` for memory IDs). The types in `hebbs-client` are tied to developer ergonomics (ULID for memory IDs, `HashMap<String, serde_json::Value>` for context). These are different concerns with different evolution pressures. Duplicating the type definitions costs a few hundred lines of code. Coupling them costs coordination overhead for the life of the project.

**Memory ID representation in the client SDK:** Memory IDs are `Ulid` type (from the `ulid` crate), not `Vec<u8>`. The client SDK converts between `Ulid` and the 16-byte proto representation at the conversion boundary. This means the developer writes `client.get(memory_id)` where `memory_id` is a `Ulid`, not `client.get(&memory_id.to_bytes())`. The ULID's `Display` implementation produces the canonical 26-character string, matching the CLI output format established in Phase 9.

### 3. Connection Management Strategy

tonic operates over HTTP/2, which supports multiplexing -- multiple concurrent RPCs over a single TCP connection. This means "connection pooling" in the traditional sense (maintaining N TCP connections) is less critical than with HTTP/1.1. However, connection lifecycle management still matters.

**Decision: single `tonic::Channel` with lazy connect, automatic reconnect, and configurable keepalive.**

The `HebbsClient` wraps a single `tonic::transport::Channel`. The channel is created at `build()` time but the TCP connection is established lazily on the first RPC call (tonic's default behavior). If the connection drops (server restart, network partition), tonic automatically reconnects on the next RPC attempt. The client adds a keepalive configuration to detect dead connections proactively rather than discovering them on the next RPC.

For high-throughput scenarios (Phase 12 benchmarks, production agents with high recall rates), a single HTTP/2 connection may become a bottleneck due to head-of-line blocking or flow control limits. The client builder accepts a `channel_count(n)` parameter that creates N channels and round-robins RPCs across them. The default is 1. This is a future extension point -- Phase 10 implements it but the default behavior is a single channel.

**Keepalive configuration defaults:**

| Parameter | Default | Rationale |
|-----------|---------|-----------|
| Keepalive interval | 30 seconds | Detect dead connections before the next RPC fails. Matches common load balancer idle timeout defaults. |
| Keepalive timeout | 10 seconds | Time to wait for keepalive ACK. 10 seconds is generous for a local or same-datacenter connection. |
| HTTP/2 initial connection window | 4MB | Larger than the 64KB default. Reduces flow control round-trips for large recall results. |
| HTTP/2 initial stream window | 2MB | Balances memory usage per stream with throughput. |

### 4. Retry Policy Design

Not all operations are safe to retry. `remember()` is not idempotent -- retrying it could create duplicate memories. `recall()` is idempotent -- retrying it produces the same results (modulo concurrent writes, which is acceptable).

**Decision: retry only on transient errors, only for idempotent operations, with configurable policy.**

**Idempotency classification:**

| Operation | Idempotent | Retryable | Rationale |
|-----------|-----------|-----------|-----------|
| `remember` | No | No | Creates a new memory with a new ULID. Retry creates a duplicate. |
| `get` | Yes | Yes | Pure read. |
| `recall` | Yes | Yes | Pure read (reinforcement update is best-effort and server-side). |
| `prime` | Yes | Yes | Composition of reads. |
| `revise` | No | No | Modifies state. Retry could re-apply a revision with stale predecessors. |
| `forget` | Conditionally | No (default) | By-ID forget is idempotent (forgetting a forgotten memory is a no-op). By-criteria forget could match different memories on retry if concurrent writes occurred. Default to non-retryable for safety. |
| `subscribe` | N/A | Reconnect | Subscribe is a long-lived stream. On disconnect, the client can re-establish the subscription, but it is not a retry -- it is a new subscription that may miss pushes during the gap. |
| `reflect` | No | No | Triggers computation with side effects (creates insights). |
| `insights` | Yes | Yes | Pure read. |

**Retryable error classification:**

| gRPC Status | Retryable | Rationale |
|-------------|-----------|-----------|
| `UNAVAILABLE` | Yes | Server temporarily unreachable (restart, load spike). |
| `RESOURCE_EXHAUSTED` | Yes (with backoff) | Rate limited. Back off and retry. |
| `DEADLINE_EXCEEDED` | Yes (once) | Timeout may be transient. Retry once with same timeout. |
| `INTERNAL` | No | Server bug. Retrying will hit the same bug. |
| `INVALID_ARGUMENT` | No | Client error. Retrying will send the same bad request. |
| `NOT_FOUND` | No | Resource does not exist. Retrying will not make it appear. |
| `ABORTED` | Yes | Transaction conflict. Retry may succeed. |

**Retry budget:** Maximum 3 attempts (1 initial + 2 retries). Exponential backoff: 50ms, 200ms, 800ms. Jitter: ±25%. These are configurable via the builder. A `RetryPolicy` struct captures the configuration. Setting `max_retries(0)` disables retry entirely.

### 5. Client Error Taxonomy

The client SDK defines its own error type rather than re-exporting `tonic::Status` or `hebbs-core::HebbsError`. This is deliberate: the client error type is part of the public API contract and must be stable across proto schema changes and tonic version upgrades.

**Decision: `ClientError` enum with variants that map to developer-actionable categories.**

| Variant | Source | When | Actionable message pattern |
|---------|--------|------|---------------------------|
| `ConnectionFailed` | Channel creation or first RPC | Server unreachable, DNS failure, TLS handshake failure | "Cannot connect to {endpoint}: {reason}" |
| `Timeout` | Client-side deadline exceeded | Operation took longer than configured timeout | "Operation timed out after {duration}" |
| `NotFound` | gRPC `NOT_FOUND` | Memory ID does not exist | "Memory {id} not found" |
| `InvalidInput` | gRPC `INVALID_ARGUMENT` | Client sent bad data (should be caught client-side first) | Server's error message passed through |
| `ServerError` | gRPC `INTERNAL` | Bug in the server | "Server error: {message}. This may be a bug in HEBBS." |
| `Unavailable` | gRPC `UNAVAILABLE` after retries exhausted | Server persistently unreachable | "Server unavailable after {n} attempts" |
| `RateLimited` | gRPC `RESOURCE_EXHAUSTED` after retries exhausted | Sustained overload | "Rate limited after {n} attempts. Back off." |
| `SubscriptionClosed` | Stream ended unexpectedly | Server closed the subscribe stream | "Subscription closed by server: {reason}" |
| `Serialization` | Proto encoding/decoding failure | Shouldn't happen with matching proto versions | "Failed to encode/decode: {reason}" |

Every variant carries structured context: which operation was attempted, which endpoint, what the timeout was, how many retries were attempted. The error implements `std::error::Error` and `Display` with actionable messages.

### 6. Subscribe Stream Design

Subscribe is the only streaming operation. The server sends `SubscribePushMessage` proto messages as a server-streaming RPC. The client must expose this as an idiomatic Rust async stream.

**Decision: `subscribe()` returns `SubscribeHandle` with an async `Stream` of `SubscribePush` plus a `feed()` method and lifecycle controls.**

The handle provides:

- `stream()` -- returns `impl Stream<Item = Result<SubscribePush, ClientError>>`. The consumer awaits pushes with `while let Some(push) = handle.stream().next().await { ... }`.
- `feed(text)` -- sends text to the subscription via the `Feed` RPC. This is a separate RPC call, not part of the stream. Returns `Result<(), ClientError>`.
- `close()` -- sends `CloseSubscription` RPC and drops the stream. After close, `stream().next()` returns `None`.

**Reconnection on subscribe:** If the stream drops due to a network error, the client does NOT automatically reconnect and re-subscribe. This is a deliberate choice. Subscribe is stateful -- the server maintains per-subscription bloom filters, centroid state, and deduplication sets. Reconnecting creates a new subscription with fresh state. The client cannot transparently resume where it left off. Instead, the stream yields a `SubscriptionClosed` error, and the consumer decides whether to re-subscribe. This is the same model as `redis-cli SUBSCRIBE` and gRPC streaming best practices.

**Backpressure:** The client does not buffer pushes beyond the gRPC transport buffer. If the consumer does not poll the stream, gRPC's HTTP/2 flow control applies backpressure to the server. This is the correct behavior -- the server's output buffer (bounded, drop-oldest from Phase 6) handles the case where the consumer is slow.

### 7. Builder Pattern Design

The client SDK uses a builder pattern for construction. This is standard Rust API design for types with many optional configuration parameters.

**Decision: `HebbsClient::builder()` returns a `ClientBuilder` with chained configuration methods, finalized by `.build().await`.**

`build()` is async because it may validate the endpoint (DNS resolution) and optionally establish the initial connection (configurable via `connect_lazy(true/false)`). Default is lazy connect (no network I/O during `build()`).

**Builder methods and their defaults:**

| Method | Type | Default | Phase 10 behavior |
|--------|------|---------|-------------------|
| `endpoint(s)` | `impl Into<String>` | `"http://localhost:6380"` | Sets the gRPC endpoint. Required for non-default. |
| `timeout(d)` | `Duration` | 30 seconds | Per-RPC deadline. |
| `connect_timeout(d)` | `Duration` | 5 seconds | TCP connection establishment timeout. |
| `retry_policy(p)` | `RetryPolicy` | 3 attempts, exponential backoff | Retry configuration for idempotent operations. |
| `channel_count(n)` | `usize` | 1 | Number of HTTP/2 channels for round-robin. |
| `connect_lazy(b)` | `bool` | `true` | Whether to defer TCP connect to first RPC. |
| `keepalive_interval(d)` | `Duration` | 30 seconds | HTTP/2 keepalive ping interval. |
| `api_key(s)` | `impl Into<String>` | None | Stored but not applied until Phase 13. |
| `tls_config(c)` | `TlsConfig` | None | Stored but not applied until Phase 13. |
| `user_agent(s)` | `impl Into<String>` | `"hebbs-client-rust/{version}"` | HTTP/2 user-agent header. |

The builder validates inputs at `build()` time, not at setter time. This means `builder.timeout(Duration::ZERO)` does not panic -- it returns `Err(ClientError::InvalidInput)` from `build()`. This is consistent with Principle 12 (validate at the boundary).

### 8. FFI Handle Pattern

The FFI must bridge Rust's ownership model to C's manual memory management. The standard pattern is an opaque handle: a raw pointer to a heap-allocated Rust value that C code holds but cannot inspect.

**Decision: opaque `HebbsHandle` as `*mut c_void` wrapping `Arc<Engine>`, plus a `HebbsResult` as `*mut c_void` wrapping allocated result data.**

The lifecycle:

1. `hebbs_open(config_json, config_len) -> *mut c_void` -- allocates an `Engine`, wraps it in `Arc`, boxes it, returns the raw pointer. Returns null on failure (error retrievable via `hebbs_last_error()`).
2. All operation functions take `handle: *mut c_void` as the first parameter. They cast it back to `&Arc<Engine>` using `unsafe` reference from raw pointer. The `Arc` is not consumed -- the handle remains valid.
3. `hebbs_close(handle: *mut c_void)` -- reconstructs the `Box<Arc<Engine>>` from the raw pointer and drops it. The `Arc` reference count decrements. If this was the last reference, the engine shuts down.

**Why `Arc<Engine>` instead of `Box<Engine>`:** The `Engine` is `Send + Sync` (RocksDB handle is `Arc<DB>`, indexes use `RwLock`). Wrapping in `Arc` allows the FFI host to share the handle across threads safely. The FFI documentation states that the handle is thread-safe and may be used from multiple threads concurrently, matching the engine's concurrency model established in Phase 1.

**Safety invariants documented per `unsafe` block:**

- The handle pointer is non-null and was returned by `hebbs_open`.
- The handle has not been passed to `hebbs_close` (use-after-free is the caller's responsibility, documented in the C header).
- String pointers passed to FFI functions are valid UTF-8 with the specified length (the FFI validates UTF-8 and returns an error if invalid, rather than invoking undefined behavior).

### 9. FFI Error Reporting

C has no `Result` type. Three patterns exist for error reporting across FFI:

| Pattern | Mechanism | Used by |
|---------|-----------|---------|
| Return codes | Function returns `int` (0 = success, negative = error category) | POSIX, SQLite |
| Out-parameter | Function takes `char **error` parameter, sets it on failure | libgit2 |
| Thread-local | `errno`-style thread-local error, retrieved by separate function | OpenSSL, POSIX `errno` |

**Decision: return codes plus thread-local error string, retrievable via `hebbs_last_error()`.**

Every FFI function returns `i32`: 0 for success, negative values for error categories. The error categories mirror the client SDK's error taxonomy:

| Return code | Meaning |
|-------------|---------|
| 0 | Success |
| -1 | Invalid argument (null pointer, invalid UTF-8, out-of-range value) |
| -2 | Not found (memory ID does not exist) |
| -3 | Storage error (RocksDB I/O failure) |
| -4 | Embedding error (ONNX model failure) |
| -5 | Internal error (invariant violation) |
| -6 | Invalid handle (null or already closed) |

On any non-zero return, the caller can retrieve a detailed error message via `hebbs_last_error() -> *const c_char`. This returns a pointer to a thread-local, null-terminated UTF-8 string that remains valid until the next FFI call on the same thread. The caller must NOT free this pointer -- it is owned by the FFI layer.

**Why thread-local over out-parameter:** Out-parameters add a parameter to every function signature and require the caller to manage the error string lifetime. Thread-local errors are simpler for the caller and match the `errno` convention that C developers already understand. The thread-local string is allocated once per thread and reused across calls, so there is no allocation pressure.

### 10. FFI Data Exchange Format

C cannot consume Rust structs directly (different layout, different memory model). The FFI must exchange structured data (memories, recall results, context maps) across the boundary.

**Decision: JSON serialization for all structured data exchange.**

Operations that return structured results (remember returns a memory, recall returns a list of results) serialize the result as a JSON string. The FFI function allocates the JSON string and returns a pointer to it. The caller reads the JSON and frees it via `hebbs_free_string()`.

Why JSON over custom C structs:

- **Simplicity.** A C struct with embedded strings, arrays of variable-length elements, and nested maps requires a tree of pointers with individual free functions. JSON is a single string.
- **Language interoperability.** Every language has a JSON parser. The FFI consumer (Python, Ruby, Java, C#) parses the JSON into native objects. No need for per-language struct binding generators beyond `cbindgen`.
- **Schema evolution.** Adding a field to the Memory struct in a future phase requires adding a field to the JSON output. No C header regeneration, no struct layout changes, no ABI break.
- **Performance is acceptable.** The FFI boundary is not the hot path -- `hebbs-core::Engine` is the hot path. The JSON serialization overhead (microseconds for a typical Memory) is negligible compared to the engine operation (milliseconds for embed + index). If profiling proves otherwise in Phase 12, the FFI can add a zero-copy binary serialization mode as an optimization, but JSON is the correct starting point for correctness and simplicity.

**Input parameters** for simple operations (remember, forget by ID) are passed as individual C function arguments (pointer + length for strings, f32 for importance). For complex operations (recall with strategy configuration, revise with context map), the input is a JSON string parameter. This keeps simple operations simple and complex operations possible.

### 11. FFI Subscribe Model

Subscribe is fundamentally a push-based pattern. In Rust, it is a background thread that pushes to a channel. In the FFI, three options exist:

| Option | Mechanism | Complexity | Thread safety |
|--------|-----------|------------|---------------|
| A: Polling | `hebbs_subscribe_poll(handle) -> *const c_char` returns the next push or null | Low | Safe (caller controls when to poll) |
| B: Callback | `hebbs_subscribe(handle, callback_fn, user_data)` invokes the callback from a Rust thread | Medium | Caller's callback must be thread-safe |
| C: File descriptor | `hebbs_subscribe(handle) -> fd`, caller selects/polls the fd, then calls `hebbs_subscribe_read()` | High | Safe (integrates with event loops) |

**Decision: callback-based model (Option B) with a polling fallback (Option A).**

Option B is the most natural for real-time push: the HEBBS subscription thread invokes the C function pointer with the push data whenever a relevant memory arrives. The callback receives a JSON-encoded push and a user-data pointer for context. The callback is invoked from a Rust-managed thread, so it must be safe to call from any thread.

Option A is provided as a simpler alternative for consumers that cannot handle cross-thread callbacks (single-threaded languages, some embedded environments). The consumer calls `hebbs_subscribe_poll()` in a loop, which returns the next push or null if none is available. This requires the consumer to manage the polling frequency.

**Callback signature:**

```
typedef void (*hebbs_subscribe_callback)(
    const char *push_json,    // null-terminated JSON string, valid only for the duration of the callback
    uint32_t push_json_len,   // length in bytes (excluding null terminator)
    void *user_data           // opaque pointer passed to hebbs_subscribe
);
```

The push JSON string is stack-allocated (on the Rust thread's stack or in a reusable buffer). It is valid only for the duration of the callback invocation. If the callback needs to keep the data, it must copy it. This avoids allocation per push.

### 12. FFI Configuration Model

Opening a HEBBS engine requires configuration: data directory, embedding model path, HNSW parameters, decay settings. In Rust, this is a struct with builder pattern. In C, this would be a struct with 20+ fields, which is brittle (adding a field is an ABI break).

**Decision: JSON configuration string passed to `hebbs_open()`.**

The JSON structure mirrors the `hebbs.toml` server configuration (same section names, same key names) but only includes the sections relevant to the engine: `storage`, `embedding`, `decay`, `hnsw`. Server-specific sections (`server`, `telemetry`) are ignored.

Minimal configuration (defaults for everything):

```
{"storage": {"data_dir": "./hebbs-data"}}
```

Full configuration with all options:

```
{
  "storage": {"data_dir": "./agent-memory", "block_cache_mb": 128},
  "embedding": {"model_path": "/models/bge-small-en-v1.5.onnx", "dimensions": 384},
  "hnsw": {"m": 16, "ef_construction": 200, "ef_search": 100},
  "decay": {"half_life_days": 30, "sweep_interval_hours": 1, "enabled": true}
}
```

**Why JSON over TOML for FFI:** The server uses TOML because it is human-readable and human-editable (config files). The FFI uses JSON because every language can construct JSON strings programmatically without a TOML library. The key names and structure are identical -- only the serialization format differs.

### 13. Crate Dependency Graphs

**`hebbs-client` dependency graph:**

```
hebbs-client
  ├── hebbs-proto       (gRPC stubs, proto message types)
  ├── tonic              (gRPC client transport)
  ├── tokio              (async runtime -- inherited from workspace)
  ├── ulid               (memory ID type for public API)
  ├── serde + serde_json (context map handling, JSON output)
  ├── thiserror          (ClientError derive)
  └── tracing            (instrumentation spans for operations)
```

`hebbs-client` does NOT depend on: `hebbs-core`, `hebbs-storage`, `hebbs-embed`, `hebbs-index`, `hebbs-reflect`, `rocksdb`, `ort`, `ureq`, or any engine-internal crate. This is the critical compile-time firewall. A project using `hebbs-client` compiles in seconds, not minutes.

**`hebbs-ffi` dependency graph:**

```
hebbs-ffi
  ├── hebbs-core         (Engine and all public types)
  ├── hebbs-storage      (StorageBackend trait, RocksDB impl)
  ├── hebbs-embed        (Embedder trait, OnnxEmbedder)
  ├── hebbs-index        (IndexManager, HnswParams)
  ├── hebbs-reflect      (LlmProvider trait -- for reflect configuration)
  ├── serde_json         (JSON config parsing, JSON result serialization)
  ├── ulid               (memory ID conversion)
  └── libc               (C type definitions for FFI boundary)
```

`hebbs-ffi` does NOT depend on: `tonic`, `tokio`, `prost`, `axum`, or any network/async crate. This is the critical runtime firewall. The FFI shared library runs without an async runtime, without a network stack, and without a server process.

### 14. Concurrency Model for FFI

The FFI must be safe to call from multiple threads simultaneously, because the engine supports concurrent operations (Phase 1, Decision 10).

**Decision: the FFI handle is `Send + Sync`. All FFI functions are reentrant. No global state except thread-local error strings.**

The `Arc<Engine>` inside the handle is `Send + Sync` (guaranteed by Rust's type system and verified by the `hebbs-core` test suite). Multiple threads can call `hebbs_remember()` concurrently through the same handle. The engine's internal locking (RocksDB for writes, `RwLock<HnswGraph>` for vector index) serializes where needed.

The thread-local error string uses `std::thread::LocalKey<RefCell<String>>`. Each thread has its own error buffer. This means `hebbs_last_error()` returns the error from the calling thread's most recent FFI call, not from another thread's call. This is the same semantics as `errno`.

**No global state.** Multiple `HebbsHandle` instances can coexist (different databases, different configurations). The FFI does not use any global mutexes, global allocators, or process-wide singletons. This enables embedding multiple independent HEBBS engines in a single process (e.g., a test harness that opens two engines to test sync behavior).

### 15. Client SDK Instrumentation

Every client operation emits a `tracing` span for distributed tracing integration. The span captures: operation name, endpoint, memory ID (where applicable), timeout, attempt number (for retries), elapsed time, outcome (success or error category).

**Decision: opt-in tracing via the `tracing` crate. No mandatory logging.**

The client SDK adds `#[instrument]` attributes to all public methods. If the consumer has a tracing subscriber (e.g., `tracing-subscriber` with OpenTelemetry export), spans are emitted. If no subscriber is registered, spans are no-ops with zero overhead (tracing's design guarantee).

This means the client SDK automatically participates in the consumer's observability infrastructure without the consumer writing any instrumentation code. It also means the client SDK never prints to stdout/stderr -- there is no `println!` debugging.

## Risk Register

| Risk | Impact | Probability | Mitigation |
|------|--------|-------------|------------|
| Client SDK domain types diverge from `hebbs-core` types over time, causing semantic mismatches | High -- a field that means one thing in the client and another in the engine is a silent data corruption path | Medium (no compile-time enforcement of type parity) | Every client domain type has a doc comment citing the corresponding `hebbs-core` type. A Phase 12 integration test creates memories via the client SDK and reads them via `hebbs-core` (same database), verifying field-for-field equivalence. A lint test in CI compares the field names of client types against core types and fails on divergence. |
| FFI `unsafe` blocks have soundness bugs (use-after-free, null dereference, data race) | Critical -- undefined behavior in consumer's process, potential security vulnerability | Medium (FFI boundary code is inherently unsafe) | Every `unsafe` block has a written safety invariant. All FFI functions validate handle non-null before dereferencing. MIRI tests (`cargo +nightly miri test`) run in CI for the FFI crate. Address sanitizer (ASAN) tests run the C test program under sanitizers. The FFI crate has a dedicated safety review checklist in its CONTRIBUTING section. |
| JSON serialization overhead at the FFI boundary becomes a bottleneck for high-frequency operations | Medium -- would degrade embedded mode performance below server mode | Low (engine operations dominate latency; JSON encoding is microseconds vs milliseconds) | Phase 12 benchmarks compare FFI-mode vs direct-engine-mode latency. If JSON overhead exceeds 10% of operation latency, introduce an opt-in binary serialization path (bitcode or MessagePack) as an alternative to JSON. |
| Subscribe callback invoked from Rust thread panics in C code, causing undefined behavior | High -- process crash with no recovery | Medium (C callbacks are unchecked) | The FFI subscribe wrapper catches panics at the FFI boundary using `std::panic::catch_unwind`. If the callback panics (shouldn't happen in C, but possible if callback calls back into Rust), the panic is caught, the subscription is closed, and `hebbs_last_error()` reports the panic message. The callback contract documented in `hebbs.h` states that the callback must not call any `hebbs_*` functions (no re-entrancy). |
| `cbindgen` generates incorrect or incomplete C header for complex Rust types | Low -- header is wrong, C consumers fail to compile | Low (cbindgen is mature and widely used) | The generated header is checked into the repository and verified by CI. A minimal C test program includes the header and calls every FFI function, compiled with `-Wall -Werror`. Any cbindgen regression breaks the C compilation test. |
| tonic channel silently drops requests under high concurrency due to HTTP/2 flow control | Medium -- client appears to hang or timeout | Low (HTTP/2 multiplexing handles typical workloads) | The client builder exposes `channel_count(n)` for distributing load across multiple connections. Phase 12 benchmarks test 1000 concurrent recall operations to verify no dropped requests. |
| Retry logic masks transient errors that should be surfaced to the caller | Medium -- caller does not know their operation failed temporarily | Medium (retry hides information) | Every `ClientError` variant includes a `retries_attempted` field. The tracing span logs each retry attempt with the error that triggered it. The caller can inspect the error to see how many retries occurred. |

## Testing Strategy

### Layer 1: Unit tests (in-crate)

**`hebbs-client`:**

- **Type conversion round-trips:** For every domain type (Memory, RecallInput, ForgetCriteria, etc.), construct a domain value, convert to proto, convert back, verify equality. This tests the conversion layer in isolation without a server.
- **Builder validation:** Verify that `build()` returns `Err` for: zero timeout, empty endpoint, negative channel count. Verify that `build()` succeeds with only default values.
- **Retry policy logic:** Test the retry decision function: given an operation and a gRPC status, verify whether retry is attempted. Test exponential backoff timing. Test jitter bounds.
- **Error mapping:** For every gRPC status code, verify the `ClientError` variant produced. Verify error messages are actionable and include context.
- **Memory ID conversion:** Verify `Ulid` to 16-byte proto and back, for random and edge-case ULIDs (minimum, maximum, zero).

**`hebbs-ffi`:**

- **Handle lifecycle:** `hebbs_open` returns non-null, `hebbs_close` succeeds, double-close does not crash (returns error code).
- **Error reporting:** After a failing FFI call, `hebbs_last_error()` returns a non-empty string. After a succeeding call, `hebbs_last_error()` returns empty or null.
- **JSON parsing:** Verify that the configuration JSON parser accepts minimal config, full config, and rejects invalid JSON, missing required fields, and out-of-range values.
- **UTF-8 validation:** Verify that passing invalid UTF-8 as content returns error code -1, not undefined behavior.
- **Thread-local isolation:** Two threads make failing FFI calls with different errors. Each thread's `hebbs_last_error()` returns its own error, not the other thread's.

### Layer 2: Property-based tests

- **Client type conversion fidelity:** For any valid `Memory` struct (generated by proptest), `to_proto(from_proto(memory)) == memory`. For any valid proto `Memory` message, `from_proto(to_proto(memory))` preserves all field values.
- **Memory ID round-trip:** For any 128-bit value, `Ulid::from_bytes(bytes).to_bytes() == bytes`. For any ULID string representation, `Ulid::from_string(s).to_string() == s`.
- **Retry policy bounds:** For any sequence of retry-eligible errors, the total number of attempts never exceeds `max_retries + 1`. The total backoff time never exceeds the sum of the geometric series.
- **FFI JSON output validity:** For any memory returned by `hebbs_remember`, the JSON output is valid JSON parseable by `serde_json::from_str`.

### Layer 3: Integration tests

**Client SDK against a running server:**

Start `hebbs-server` with in-memory storage backend (no RocksDB, fast startup) as a test fixture. Run the following test sequences:

- **Full lifecycle:** `remember` -> `get` -> `recall` (all strategies) -> `revise` -> `recall` (verify update) -> `forget` -> `get` (verify not found). All operations via `hebbs-client`.
- **Concurrent operations:** 10 tokio tasks each performing 100 remember + recall cycles simultaneously. All 1000 memories must be recallable afterward.
- **Subscribe stream:** Subscribe, remember a matching memory, verify the push arrives on the stream, close the subscription, verify the stream ends.
- **Retry behavior:** Configure a proxy that drops 50% of requests. Verify that idempotent operations (recall, get) eventually succeed within the retry budget. Verify that non-idempotent operations (remember) fail immediately on first error.
- **Timeout behavior:** Configure a 100ms timeout. Send a recall against 100K memories (takes >100ms on in-memory backend). Verify the client returns `Timeout` error, not a hang.
- **Connection recovery:** Start server, connect client, stop server, attempt an operation (verify error), restart server, attempt an operation (verify success via auto-reconnect).

**FFI against embedded engine:**

- **Full lifecycle (C test program):** A minimal C program that includes `hebbs.h`, calls `hebbs_open`, `hebbs_remember`, `hebbs_recall`, `hebbs_close`, and verifies return codes. Compiled with `-Wall -Werror -fsanitize=address`.
- **Full lifecycle (Rust FFI tests):** The same sequence but called from Rust test code using `unsafe` FFI calls. This tests the FFI without requiring a C compiler in CI.
- **Concurrent access:** 4 threads sharing a handle, each performing 100 remember + recall operations. Verify no data corruption and no data races (TSAN).
- **Error cases:** Open with invalid config JSON, remember with null content pointer, get with non-existent ID. Verify error codes and `hebbs_last_error()` messages.

### Layer 4: Parity tests

A parity test suite runs the same logical test sequence against both the client SDK (server mode) and the FFI (embedded mode). The test asserts that the results are identical: same memories created, same recall results returned, same error behavior. This ensures the two access modes are behaviorally equivalent.

### Layer 5: Criterion benchmarks

- **Client SDK round-trip latency:** `remember` + `get` via client SDK against a local server. Measures the overhead of proto conversion + gRPC transport + tonic channel. Compared against raw tonic client (no SDK wrapper) to isolate SDK overhead.
- **FFI round-trip latency:** `hebbs_remember` + `hebbs_recall` via FFI. Measures the overhead of JSON serialization + FFI boundary crossing. Compared against direct `Engine::remember` + `Engine::recall` to isolate FFI overhead.
- **Type conversion:** Proto-to-domain and domain-to-proto conversion for a typical `Memory` struct. Target: < 1µs per conversion.
